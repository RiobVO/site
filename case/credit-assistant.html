§ 01БрифBrief

Банк выдаёт кредиты SME — малому и среднему бизнесу. У кредитного комитета на каждую заявку уходит около двух дней: аналитик собирает финотчётность клиента в Excel, считает 6 – 8 коэффициентов, формирует pdf-отчёт, отправляет на следующего аналитика. Если по дороге кто-то ошибся в формуле — возврат на круг. The bank issues SME loans. Each application takes the credit committee about two days: an analyst collects the client's financial statements in Excel, calculates 6–8 ratios, generates a PDF report, and passes it to the next analyst. If anyone makes a formula mistake along the way — back to square one.

Задача: заменить эту цепочку Excel-файлов одним сервисом, который принимает структурированную форму заявки, прогоняет через 24 правила (ЦБ РУз, Basel III, IFC — лайтовая адаптация), и выдаёт двуязычный PDF — РУ + Узб — c явным аудит-trail каждого решения. Goal: replace this chain of Excel files with a single service that accepts a structured application form, runs it through 24 rules (Central Bank of Uzbekistan, Basel III, IFC — light adaptation), and produces a bilingual PDF — RU + UZ — with an explicit audit trail for every decision.

Жёсткие ограничения: банковский on-premise (никакого облака), LDAP для аутентификации, MFA для роли «утверждающий», полный аудит для регулятора, OWASP ASVS L2 как минимум. Hard constraints: bank's on-premise infrastructure (no cloud), LDAP authentication, MFA for the "approver" role, full audit trail for the regulator, OWASP ASVS L2 minimum.

§ 02АрхитектураArchitecture

Один сервис, четыре слоя. Clean Architecture без фанатизма — ровно столько границ, сколько нужно, чтобы завтра можно было заменить ReportLab на WeasyPrint, а LDAP на Keycloak без переписи доменного слоя. One service, four layers. Clean Architecture without fanaticism — just enough boundaries to swap ReportLab for WeasyPrint, or LDAP for Keycloak tomorrow without rewriting the domain layer.

Рис. 1 · Четыре слоя. Стрелки направлены только внутрь — к домену.Fig. 1 · Four layers. Arrows point inward only — toward the domain.

Главная инвестиция — в domain. ScoringEngine принимает Application, возвращает Decision с полным trail применённых правил. Pure-функция: никаких IO, никаких datetime.now(), никаких глобалов. Это значит, что весь домен покрывается unit-тестами без единого мока. The main investment is in the domain. ScoringEngine takes an Application, returns a Decision with a full trail of applied rules. Pure function: no IO, no datetime.now(), no globals. This means the entire domain is unit-testable without a single mock.

24 правила, один движок24 rules, one engine

Каждое правило — класс с id, source (ЦБ РУз / Basel / IFC), applies_to и evaluate(app) -> RuleResult. Правила версионируются: когда регулятор меняет коэффициент, добавляется R-12.v2, старая версия остаётся для ретроспективы. Each rule is a class with id, source (CBU / Basel / IFC), applies_to, and evaluate(app) -> RuleResult. Rules are versioned: when the regulator changes a ratio, R-12.v2 is added while the old version stays for retrospective analysis.

domain/rules/r17_dscr.py basel iii 
class DebtServiceCoverageRatio(Rule):
    id = "R-17"
    source = RuleSource.BASEL_III
    threshold = Decimal("1.25")
    severity = Severity.HARD_FAIL

    def evaluate(self, app: Application) -> RuleResult:
        if app.financials.ebitda <= 0:
            return RuleResult.fail("ebitda_non_positive")

        dscr = app.financials.ebitda / app.requested.annual_debt_service
        if dscr < self.threshold:
            return RuleResult.fail(
                code="dscr_below_1_25",
                measured=dscr,
                threshold=self.threshold,
            )
        return RuleResult.passed(measured=dscr)

Движок прогоняет правила в топологическом порядке (некоторые зависят от результата других — например, проверка по залогу применяется только если DSCR ниже порога), собирает DecisionTrail и отдаёт наверх. PDF-генератор читает trail и превращает в разделы отчёта — по каждому правилу видно, что именно сработало. The engine runs rules in topological order (some depend on others — e.g., collateral check only applies if DSCR is below the threshold), collects a DecisionTrail, and passes it up. The PDF generator reads the trail and turns it into report sections — every triggered rule is explicitly visible.

§ 03Что было сложноChallenges

1. Двенадцать форматов входных таблиц1. Twelve input spreadsheet formats

Финотчётность приходит из 1С, из Excel, иногда из pdf со сканированной таблицы. На старте казалось, что найдём 2 – 3 формата и закроем. На неделе 3 их было уже двенадцать. Financial statements come from 1C, Excel, and sometimes from PDFs with scanned tables. At the start, we expected 2–3 formats. By week 3, there were already twelve.

Решение — слой FinancialIngestor: цепочка парсеров, каждый отвечает за один формат, плюс ManualEntry как дефолт. Pydantic v2 на границе — дальше в домен заходит только валидный Financials. Никаких dict[str, Any] внутри. Solution — a FinancialIngestor layer: a chain of parsers, each responsible for one format, plus ManualEntry as the default. Pydantic v2 at the boundary — only a validated Financials object enters the domain. No dict[str, Any] inside.

2. Bilingual PDF без кошмара2. Bilingual PDF without the nightmare

Отчёт двуязычный: каждая страница содержит русскую и узбекскую версии в двух колонках. Узбекский — кириллица и латиница, по выбору банка. ReportLab напрямую с кириллицей не дружит без шрифтов; WeasyPrint умеет HTML/CSS, но медленный. The report is bilingual: every page has Russian and Uzbek versions in two columns. Uzbek — Cyrillic or Latin script, the bank's choice. ReportLab doesn't handle Cyrillic well without custom fonts; WeasyPrint supports HTML/CSS but is slow.

Гибрид: WeasyPrint для титулки и структуры, ReportLab для таблиц с коэффициентами, pypdf чтобы сшить. p95: 748 мс. Было 2.3 секунды до разделения. Hybrid approach: WeasyPrint for the title page and structure, ReportLab for ratio tables, pypdf to merge. p95: 748 ms. Was 2.3 seconds before the split.

90% сложности оказалось не в банковской математике, а в том, чтобы PDF был одинаковый на Linux-сервере банка и на моём ноутбуке. 90% of the complexity turned out to be not in the banking math, but in making the PDF look identical on the bank's Linux server and on my laptop.

3. On-premise air-gapped3. On-premise air-gapped

Сервер банка не ходит в интернет. Пакеты, зависимости, образ Docker, всё — собирается на CI и приезжает на флешке (буквально). Это значит: vendored deps, проверка SBOM на каждом релизе, отдельный playbook для upgrade без интернета. The bank's server has no internet access. Packages, dependencies, Docker image — everything is built on CI and delivered via USB drive (literally). This means: vendored deps, SBOM verification on every release, a dedicated playbook for upgrades without internet.

§ 05Что я вынесLessons learned

(а) Pydantic v2 на границе — это не «валидация», это договор между слоями. Сначала боролся, потом осознал, потом полюбил. (a) Pydantic v2 at the boundary is not "validation" — it's a contract between layers. First I resisted, then I understood, then I embraced it.

(б) Версионируемые правила — дорого на старте, бесценно через полгода, когда регулятор поменял один коэффициент и нужно объяснить, почему прошлогодние решения были правильными по тогдашним правилам. (b) Versioned rules are expensive at the start, priceless six months later when the regulator changes a ratio and you need to explain why last year's decisions were correct under the rules at the time.

(в) «On-premise» на деле означает «у вас не будет интернета, и точно сломается то, что вы не могли предусмотреть без опыта on-prem». На следующем проекте я закладывал отдельные две недели на это по умолчанию. (c) "On-premise" in practice means "you will have no internet, and something you couldn't anticipate without on-prem experience will definitely break." On the next project, I budgeted two extra weeks for this by default.
